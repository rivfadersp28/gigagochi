# ТЗ: движок заготовок персонажей на основе PokéAPI

Дата: 2026-07-05

Статус: только техническое задание. Код в рамках этого документа не меняется.

## 1. Цель

Подключить режим интерфейса `Использовать заготовки` к описаниям покемонов,
полученным из PokéAPI, и отключить текущий источник заготовок на базе локальных
Character Card шаблонов.

Целевой эффект:

- пользователь вводит произвольный промпт персонажа;
- приложение генерирует визуального персонажа по пользовательскому промпту;
- если включен режим `Использовать заготовки`, бэкенд выбирает наиболее
  подходящую запись из базы описаний PokéAPI;
- выбранное описание превращается в скрытый источник характера, лора, манеры
  общения, привычек и деталей поведения;
- прямые упоминания Pokémon, PokéAPI, исходных имен и эволюционных названий не
  попадают в пользовательский опыт, промпты генерации и ответы персонажа;
- возрастные модификаторы приложения `малой`, `подросток`, `взрослый` продолжают
  влиять на стиль общения и имеют приоритет над источником-заготовкой.

## 2. Текущий контекст

Найденная цепочка в приложении:

- `frontend/src/components/CreatePetForm.tsx` содержит переключатель
  `Использовать заготовки` и состояние `useTemplatePresets`.
- `frontend/src/lib/api.ts` отправляет `useTemplatePresets` в запрос генерации.
- `backend/app/routers/tma.py` прокидывает флаг в сервис генерации.
- `backend/app/services/image_service.py` при `use_template_presets=True`
  вызывает `create_character_bible_from_template(description)`.
- `backend/app/services/character_templates.py` сейчас загружает заготовки из
  `backend/data/character_templates` и адаптирует локальные Character Card.
- `backend/app/services/pet_reply_engine/age_profiles.py` уже содержит профили
  возраста `baby`, `teen`, `adult` с русскими интерфейсными метками `малой`,
  `подросток`, `взрослый`.
- `backend/app/services/pet_reply_engine/prompt_builder.py` уже добавляет
  возрастный профиль в runtime-слой промпта.

Вывод: транспорт флага и возрастный слой уже существуют. Нужно заменить источник
и адаптер режима заготовок, не переписывая всю генерацию персонажей.

- JSONL можно использовать как вспомогательный источник для тестов,

- Текущий файл содержит 100 записей. Реализация не должна иметь лимит 100 и
  должна корректно работать с полным файлом, если позже будут скачаны все
  доступные записи PokéAPI.

## 4. Формат парсинга Markdown

Парсер должен читать структуру:

```md
## Bulbasaur

- PokeAPI species ID: `1`
- Generation: `generation-i`
- Genus: Seed Pokémon
- Evolution: bulbasaur -> ivysaur -> venusaur

Descriptions:
- ...
```

На выходе нужен нормализованный внутренний объект:

```json
{
  "source_id": "1",
  "source_name": "bulbasaur",
  "display_name": "Bulbasaur",
  "generation": "generation-i",
  "genus": "Seed Pokémon",
  "evolution_path": ["bulbasaur", "ivysaur", "venusaur"],
  "descriptions": ["..."],
  "source_path": "internal://description_presets/species_descriptions.md"
}
```

Требования к парсеру:

- сохранять порядок записей;
- удалять дубли описаний после нормализации пробелов и переносов;
- поддерживать пустое или отсутствующее поле `Evolution`;
- поддерживать ветвящиеся эволюции, если формат будет расширен;
- не падать из-за нестандартных символов `POKéMON`, `Pokémon`, soft hyphen,
  лишних переносов и разных регистров;
- кэшировать результат в памяти процесса после первого чтения;
- при ошибке парсинга давать понятный лог с номером блока или заголовком записи.

## 5. Отключение текущих заготовок

При `useTemplatePresets=True` больше нельзя использовать
`backend/data/character_templates` как основной источник.

Нужное поведение:

- текущий UI-текст `Использовать заготовки` остается без изменений;
- флаг `useTemplatePresets` включает только PokéAPI-заготовки;
- старый импорт Character Card шаблонов не вызывается в этом режиме;
- старые тесты про локальные template presets нужно заменить или переписать под
  новый источник;
- при недоступном Pokémon-датасете нельзя молча возвращаться к старым
  Character Card заготовкам.

Fallback при ошибке данных:

- предпочтительно создать обычный `character_bible` без заготовки и пометить
  `provenance.preset_error`;
- старые Character Card шаблоны не использовать как скрытый fallback, потому что
  пользователь ожидает режим только на базе покемон-описаний.

## 6. Матчинг промпта с описанием

Движок выбора должен найти наиболее подходящую запись из всего доступного списка.

Вход:

- исходное пользовательское описание;
- опционально извлеченное имя пользователя для персонажа, если оно есть;
- все нормализованные записи из Markdown.

Выход:

```json
{
  "record": "...",
  "score": 0.0,
  "confidence": "low | medium | high",
  "matched_terms": ["fire", "lizard"],
  "selection_reason": "internal debug only"
}
```

MVP-алгоритм:

- нормализовать русский и английский текст;
- выделить ключевые слова про стихию, тело, среду, темперамент, поведение,
  размер, роль и настроение;
- использовать словарь русско-английских алиасов:
  `огонь/fire`, `вода/water`, `электричество/electric`, `растение/plant`,
  `камень/rock`, `лед/ice`, `призрак/ghost`, `дракон/dragon`,
  `мышь/mouse`, `ящер/lizard`, `черепаха/turtle`, `птица/bird`,
  `змея/snake`, `ночь/night`, `солнце/sun`, `лес/forest`;
- считать совпадения по `genus`, `descriptions`, обобщенным мотивам и
  эволюционной структуре;
- не отдавать бонус за исходное имя покемона в финальном пользовательском тексте;
- при низкой уверенности все равно выбирать лучшую Pokémon-запись, но
  использовать ее только как мягкий источник настроения и привычек, без переноса
  анатомии.

Будущее улучшение:

- добавить embeddings/reranker для русского промпта и английских описаний;
- хранить индекс признаков рядом с датасетом;
- покрыть selection snapshots тестами, чтобы обновление датасета не ломало
  ожидаемый выбор.

## 7. Санитизация бренда и исходных имен

Главное правило: пользователь не должен видеть, что заготовка взята из
конкретного покемона.

Запрещенные к утечке токены:

- `Pokémon`, `Pokemon`, `POKéMON`, `PokéAPI`, `PokeAPI`;
- `Poké Ball`, `POKé BALL`, `trainer`, если они используются как прямой
  франшизный контекст;
- заголовок выбранной записи, например `Bulbasaur`;
- все имена из `Evolution`, например `bulbasaur`, `ivysaur`, `venusaur`;
- имена из других полей источника, если они будут добавлены позже.

Где эти токены не должны появляться:

- в `identity.name`, `identity.species`, `identity.one_liner`;
- в `signature`, `personality`, `scenario`, `system_prompt`,
  `post_history_instructions`;
- в `character_book.entries[*].content`;
- в image prompt;
- в chat prompt;
- в ответах персонажа;
- в публичном API-ответе фронтенду.

Где можно хранить исходные данные:

- только во внутренней provenance/debug-структуре, которая не отправляется в LLM
  и не показывается пользователю;
- пример: `extensions.preset_source_internal`;
- при сериализации в промпты эта секция должна быть удалена.

Если пользователь сам явно ввел имя, совпадающее с исходным именем покемона,
имя пользователя считается пользовательским вводом, но система все равно не
должна говорить, что персонаж является покемоном или взят из PokéAPI.

## 8. Адаптация описания в Character Bible

Выбранная Pokémon-запись не должна копироваться в персонажа дословно. Она должна
быть преобразована в нейтральное, оригинальное ядро персонажа.

Поля, которые нужно формировать:

- `identity`
  - имя из пользовательского ввода или существующей логики приложения;
  - вид/роль из пользовательского промпта;
  - короткий `one_liner` без Pokémon-терминов.
- `personality`
  - 4-8 черт характера, извлеченных из описаний;
  - страхи, желания, привычки, способ привязываться к пользователю.
- `lore`
  - нейтральная история происхождения;
  - среда, привычные места, слабости и сильные стороны;
  - без прямого копирования мира Pokémon.
- `voice`
  - ритм речи;
  - любимые речевые ходы;
  - запреты на фразы, которые палят источник.
- `character_book`
  - 3-6 коротких lore entries;
  - ключи на русском и английском, если это помогает retrieval;
  - контент только в санитизированном виде.
- `visual_alignment`
  - пользовательский промпт имеет приоритет над анатомией источника;
  - анатомические детали источника используются только если не конфликтуют с
    пользовательским запросом.

Пример адаптации:

- источник: огненный ящер с пламенем как показателем жизненной силы;
- пользовательский промпт: `маленький рыцарь с горячим характером`;
- допустимый результат: вспыльчивый, теплый, храбрый персонаж, которому важно
  сохранять внутренний огонь и тренироваться;
- недопустимый результат: персонаж говорит, что он Charmander, Pokémon или что
  у него обязательно хвост с каноническим пламенем, если пользователь просил
  рыцаря.

## 9. Рост и возрастные модификаторы

В приложении уже есть возрастные стадии:

- `малой` (`baby`);
- `подросток` (`teen`);
- `взрослый` (`adult`).

Требование: возраст приложения остается runtime-слоем и всегда сильнее возраста
или стадии из источника.

Как использовать evolution chain:

- `Evolution` не показывается пользователю и не попадает в промпт именами;
- количество стадий и мотив роста можно превратить в абстрактную арку развития;
- для трехступенчатой цепочки:
  - `малой`: простые желания, любопытство, зависимость от заботы;
  - `подросток`: усиление главного мотива, больше упрямства и экспериментов;
  - `взрослый`: уверенность, стабильность, зрелое владение своей силой;
- для двухступенчатой цепочки:
  - `малой` и `подросток` могут делить первую фазу с разной уверенностью;
  - `взрослый` получает завершенную форму мотива;
- для одиночной записи:
  - рост выводится из описаний, а не из отсутствующей эволюции.

Что должно попадать в Character Bible:

```json
{
  "growth_arc": {
    "baby": "санитизированный мотив ранней стадии",
    "teen": "санитизированный мотив переходной стадии",
    "adult": "санитизированный мотив зрелой стадии"
  }
}
```

Что должно решаться в chat runtime:

- конкретная манера речи;
- длина ответа;
- степень наивности, дерзости или зрелости;
- допустимые игровые действия для текущего возраста.

## 10. Интеграция с генерацией изображения

Текущая генерация изображения должна продолжать строиться вокруг пользовательского
описания.

Правила:

- пользовательский промпт определяет внешний вид;
- Pokémon-заготовка добавляет только совместимые детали характера, настроения,
  позы, энергии и мелкие мотивы;
- если источник конфликтует с пользователем, источник ослабляется;
- image prompt не должен содержать исходных имен покемонов и слова Pokémon;
- нельзя просить модель нарисовать конкретного канонического покемона;
- если пользователь сам просит конкретную известную франшизу, это отдельный
  policy/product case и не должно быть следствием режима заготовок.

## 11. Интеграция с чатом

Чат должен использовать адаптированную заготовку как слой личности, но не как
замену runtime-состояния питомца.

Порядок приоритетов:

1. Безопасность, системные ограничения и формат ответа.
2. Текущий runtime: возраст, настроение, статы, память, состояние игры.
3. Пользовательский промпт и выбранное пользователем имя.
4. Адаптированная Pokémon-заготовка.
5. Общие fallback-дефолты.

Требования к ответам:

- персонаж не сообщает, что он покемон;
- персонаж не называет исходное имя выбранной записи;
- персонаж не ссылается на PokéAPI, тренеров, Poké Ball или канонический мир;
- характер, привычки и лор должны ощущаться как устойчивые от ответа к ответу;
- возрастный стиль должен быть заметен:
  - `малой`: проще, короче, эмоциональнее;
  - `подросток`: больше самостоятельности, реакции и пробования границ;
  - `взрослый`: спокойнее, увереннее, связнее.

## 12. API и модель данных

Публичный контракт `useTemplatePresets` менять не нужно.

Внутренний provenance:

```json
{
  "provenance": {
    "source": "description_preset",
    "source_urls": ["internal://description_presets/species_descriptions.md"],
    "license_notes": "derived from local species descriptions; do not expose source names in generated character"
  },
  "extensions": {
    "preset_source_internal": {
      "source_id": "4",
      "selection_score": 0.82,
      "confidence": "high",
      "matched_terms": ["fire", "lizard"]
    }
  }
}
```

Ограничение: `extensions.preset_source_internal` не должен попадать в
LLM-facing prompt и публичный UI. Исходные имена не должны храниться даже в
публичном `character_bible`; допустимы только `source_id`, score и служебные
метрики выбора.

## 13. Тесты

Минимальные unit-тесты:

- парсер извлекает 100 записей из текущего MD и не теряет описания;
- парсер корректно читает `PokeAPI species ID`, `Generation`, `Genus`,
  `Evolution`, `Descriptions`;
- выбор для `огненный ящер` отдает огненную ящероподобную запись;
- выбор для `электрическая мышь` отдает электрическую мышиную запись, если она
  есть в датасете;
- выбор для `черепаха с панцирем` отдает черепашью или защитную запись;
- при `useTemplatePresets=True` старый каталог `backend/data/character_templates`
  не читается;
- видимый `character_bible` не содержит Pokémon, PokeAPI, исходного имени и имен
  evolution chain;
- скрытая provenance содержит source id и score;
- image prompt не содержит запрещенных токенов;
- chat prompt не содержит запрещенных токенов;
- возраст `baby/teen/adult` добавляет правильный runtime-профиль и не
  переписывается источником.

Интеграционные тесты:

- `/tma/generate-pet` с `useTemplatePresets=True` возвращает asset set и
  `character_bible.provenance.source = "description_preset"`;
- `/tma/generate-pet` с `useTemplatePresets=False` работает как раньше без
  Pokémon-заготовок;
- созданный персонаж отвечает в чате с адаптированным характером;
- смена стадии на `малой`, `подросток`, `взрослый` меняет стиль ответа, но
  сохраняет базовый характер;
- snapshot-тест проверяет отсутствие запрещенных брендинговых токенов во всех
  текстовых полях, которые уходят в LLM и UI.

## 14. Acceptance Criteria

Фича считается готовой, если:

- переключатель `Использовать заготовки` включает новый Pokémon-based preset
  engine;
- старые Character Card заготовки не используются в этом режиме;
- из Markdown выбирается наиболее подходящая запись по пользовательскому
  промпту;
- выбранная запись превращается в оригинальный характер и лор без прямых
  Pokémon-упоминаний;
- пользовательский промпт и имя имеют приоритет над источником;
- возрастные стадии `малой`, `подросток`, `взрослый` заметно влияют на стиль
  общения;
- ни изображение, ни чат, ни публичный API не раскрывают исходного покемона;
- тесты покрывают парсинг, выбор, санитизацию, prompt output и age override.
