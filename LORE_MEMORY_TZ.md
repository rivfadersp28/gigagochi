# ТЗ: легкая лор-память персонажа

## 1. Цель

Сделать персонажа заметно интереснее без тяжелой memory-платформы:

- при создании питомца генерировать достаточно богатый, случайно варьируемый стартовый лор;
- считать стартовый лор неизменяемой базой персонажа;
- разрешить персонажу в чате аккуратно придумывать небольшие дополнительные лор-факты;
- сохранять такие факты в копилку знаний и использовать их в будущих ответах;
- не строить сложную систему сюжетного развития, квестов, прогрессии мира или многоуровневого RAG.

Формула фичи: стартовый `characterBible.lore` задает мир, дом, происхождение, отношения и голос. Во время чата персонаж может добавлять только маленькие непротиворечивые факты: имя друга, привычку родственника, деталь дома, локальную традицию, прозвище, маленький предмет, безопасную историю из прошлого.

## 2. Что берем из двух библиотек

### `companion-app-main`

Берем не код приложения целиком, а продуктовый паттерн:

- постоянная база персонажа отдельно от истории диалога;
- seed-стиль ответа через стартовые примеры;
- выбор релевантных деталей лора под текущий вопрос;
- бэкграунд персонажа как материал для ответов, а не как текст, который надо пересказывать полностью.

Не переносим:

- Clerk;
- Twilio;
- Upstash Redis;
- Pinecone/Supabase setup;
- отдельное Next.js companion-приложение.

### `mem0-main`

Берем не OpenMemory целиком, а memory-паттерн:

- add-only накопление фактов;
- разделение памяти по типам и scope;
- извлечение только важных фактов, а не каждого сообщения;
- metadata/category-подход: `canon`, `relationship`, `user_fact`, `friend_fact`, `home_fact`;
- правило: память сначала фильтруется backend resolver-ом, затем попадает в prompt.

Не подключаем в первой итерации:

- hosted Mem0;
- OpenMemory UI;
- Qdrant/Chroma/Pinecone как отдельную инфраструктуру;
- отдельный memory server.

Причина: в проекте уже есть доменная память `pet_memory`; для текущей цели достаточно усилить ее, а не заменять.

## 3. Термины

| Термин | Определение |
| --- | --- |
| `characterBible` | JSON-профиль персонажа, создается один раз при генерации питомца. |
| Стартовый лор | `characterBible.lore`: мир, дом, происхождение, связи, внутренние привычки, голос, story seeds. |
| Лор-база | Неизменяемый стартовый лор. Не перезаписывается во время чата. |
| Лор-факт | Короткое дополнительное знание, придуманное или подтвержденное в чате. |
| Копилка лора | `pet.memory.canon`: список принятых лор-фактов, которые добавляются поверх стартового лора. |
| Memory candidate | Предложение модели сохранить новый факт. Backend решает, сохранять или отклонить. |
| Canon fact | Принятый факт, который можно использовать в будущих ответах. |

## 4. Объем первой итерации

### Входит

- Улучшить генерацию стартового `characterBible.lore`.
- Добавить легкий случайный `lore_seed`, который влияет на мир, роли, открытые темы и возможных друзей.
- Оставить стартовый лор неизменяемым после создания персонажа.
- Использовать существующий `PetMemoryStateV1.canon` как копилку дополнительных лор-фактов.
- Разрешить чату возвращать `memoryCandidates` для новых лор-фактов.
- Сохранять только непротиворечивые факты через текущий resolver.
- Подмешивать сохраненные факты в prompt будущих ответов.
- Сохранить backward compatibility с `loreMemoriesToSave`, но считать его legacy.

### Не входит

- Сюжетная прогрессия мира.
- Автоматическое изменение характера.
- Переписывание `characterBible.lore` после создания.
- Сложная графовая память.
- Векторная база на первом шаге.
- Пользовательский экран редактирования памяти.
- Модераторский UI памяти.
- Миграция всего persisted chat route на новую систему, если задача касается только текущего TMA/local flow.

## 5. Продуктовое поведение

### Создание персонажа

При создании питомца backend должен генерировать:

- визуальную идею;
- личность;
- стартовый лор;
- открытые story seeds для будущих ответов;
- 2-4 потенциальные социальные роли вокруг персонажа: друг, родственник, наставник, сосед, соперник, хранитель, мастер, старший, младший.

Лор должен быть конкретным, но не перегруженным. Он должен отвечать на вопросы:

- где персонаж живет;
- как устроен его маленький мир;
- кто обычно рядом;
- чего он хочет;
- чего он боится;
- как его физическая особенность влияет на поведение;
- какие детали можно раскрывать позже.

Стартовый лор не должен быть списком случайных происшествий. Лучше задавать reusable context:

- место;
- рутину;
- роли;
- напряжение;
- незакрытые вопросы;
- физические предметы;
- социальные отношения.

### Чат

Персонаж отвечает как обычно, но если вопрос пользователя явно касается лора, он может:

- использовать стартовый `characterBible.lore`;
- использовать сохраненные `canon` факты;
- придумать одну маленькую новую деталь, если в базе есть открытый hook;
- вернуть эту деталь как `memoryCandidates`.

Пример:

Пользователь: `а кто твой лучший друг?`

Если в лоре есть только роль `друзья из мастерской`, питомец может ответить:

`мой лучший друг - Пим, маленький ключик из соседней ячейки. он всегда звенит, когда я начинаю волноваться.`

И вернуть candidate:

```json
{
  "type": "friend_fact",
  "text": "Лучший друг питомца - Пим, маленький ключик из соседней ячейки, который звенит, когда питомец волнуется.",
  "importance": 0.72,
  "confidence": 0.78
}
```

В следующих ответах питомец должен помнить Пима и не придумывать другого лучшего друга для того же вопроса.

## 6. Правила лора

### Стартовый лор

Стартовый лор создается один раз и хранится в `characterBible.lore`.

Запрещено после создания менять:

- вид персонажа;
- основной мир;
- дом;
- базовое происхождение;
- постоянные визуальные признаки;
- базовый темперамент;
- уже заданные центральные отношения.

Допустимо добавлять:

- имя уже подразумеваемого друга;
- прозвище;
- бытовую деталь дома;
- локальную традицию;
- маленький предмет;
- привычку друга;
- уточнение роли родственника;
- короткий факт о прошлом, если он не меняет origin.

### Лор-факты из чата

Лор-факт должен быть:

- одним коротким фактом;
- на русском;
- не длиннее 500 символов;
- связан со стартовым лором или уже сохраненной копилкой;
- пригоден для повторного использования;
- не временной эмоцией текущего ответа.

Не сохранять:

- настроение текущего момента;
- пересказ всего ответа;
- технические сведения о модели, prompt, API, frontend/backend, JSON, state;
- персональные чувствительные данные пользователя;
- факты, которые меняют мир, дом, вид или основные отношения;
- случайные большие события вроде спасений, войн, переездов, смерти, травм, если они не были заложены в стартовом лоре.

## 7. Минимальная архитектура

### Источник истины

Для первой итерации:

- стартовый лор: `characterBible.lore`;
- накопленные лор-факты: `LocalPetState.memory.canon`;
- факты о пользователе: `LocalPetState.memory.relationship.userFacts`;
- отношения и мягкая динамика: существующие `relationship`, `development`, `threads`, `activeGoals`.

Не добавлять новые таблицы и внешние сервисы для TMA/local flow.

### Поток данных

1. Пользователь создает питомца.
2. Backend генерирует `lore_seed`.
3. `lore_seed` добавляется в prompt генерации `characterBible`.
4. OpenAI возвращает `characterBible` со стартовым `lore`.
5. Frontend сохраняет питомца и пустую `memory`.
6. Пользователь пишет в чат.
7. Backend строит `memory_context` из стартового лора и `memory.canon`.
8. Модель отвечает и может вернуть `memoryCandidates`.
9. Backend resolver фильтрует candidates.
10. Backend возвращает `memoryPatch`.
11. Frontend применяет patch к localStorage.
12. Следующие ответы используют сохраненные факты.

## 8. Изменения в генерации персонажа

### `lore_seed`

Добавить легкий seed-объект на backend до вызова генерации `characterBible`.

Пример структуры:

```json
{
  "setting_tone": "маленькое ремесленное место",
  "social_shape": "есть наставник, сосед и один потенциальный друг",
  "background_tension": "персонаж хочет быть полезным, но боится испортить важную вещь",
  "future_reveal": "позже можно раскрыть прозвище друга или странную местную традицию"
}
```

Seed должен:

- выбираться случайно из небольших curated списков;
- не быть видимым пользователю напрямую;
- не ломать пользовательское описание;
- помогать разнообразить лор между одинаковыми prompt-ами;
- попадать только в prompt генерации `characterBible`.

Минимальный набор списков:

- `setting_tone`: мастерская, бюро находок, маленькая станция, кладовая, подземная школа, крыша, ночная пекарня, ящик путешественника, тихий причал, кристальная комната;
- `social_shape`: один друг и строгий наставник, шумные соседи, старший родственник и младший приятель, соперник и заботливый хранитель, маленькая команда помощников;
- `background_tension`: боится ошибиться, хочет доказать самостоятельность, прячет редкий звук/предмет, не любит резкие перемены, хочет найти свое место;
- `future_reveal`: прозвище, местная традиция, любимый предмет, старый спор, скрытый уголок, роль родственника.

### Prompt `characterBible`

Усилить требования:

- `story_seeds` должны быть именно открытыми hooks, а не готовыми событиями;
- `relationships.friends` может содержать роли без точных имен;
- если имя друга не задано в стартовом лоре, модель может придумать его позже в чате;
- `lore` должен давать пространство для будущих мелких уточнений.

Не нужно добавлять новые обязательные поля в schema, если текущих `lore`, `relationships`, `story_seeds` достаточно.

## 9. Изменения в chat prompt

Prompt ответа уже содержит блоки:

- `Лор питомца`;
- `Закрепленная память лора`;
- `Память канона`;
- `Память отношений`;
- `Открытые темы`;
- `Выводы`;
- `Текущие желания`;
- `Развитие`.

Нужно уточнить правила:

- стартовый лор важнее chat-added facts при конфликте;
- chat-added facts важнее новой импровизации;
- если факт уже есть в `canon`, не придумывать альтернативу;
- если пользователь спрашивает неизвестную деталь, можно придумать только одну маленькую деталь;
- если придумана устойчивая деталь, обязательно предложить `memoryCandidates`;
- если ответ не про лор, не добавлять лор-факт без необходимости.

## 10. Изменения в resolver

Использовать текущий resolver как основной фильтр.

Проверить и при необходимости усилить:

- reject технических фактов;
- reject sensitive facts;
- reject `canon_home_conflict`;
- reject `canon_species_conflict`;
- reject крупных изменений мира;
- dedupe по normalized key;
- reinforcement существующего факта вместо дубля.

Новые или уточненные правила:

- для `friend_fact`, `family_fact`, `home_fact`, `origin_fact`, `world_fact`, `preference_fact`, `fear_fact`, `habit_fact`, `voice_fact` сохранять только если нет конфликта со стартовым `characterBible`;
- если в тексте есть маркеры `теперь живет в другом месте`, `переехал`, `стал другим существом`, `раньше этого не было`, `забыл свой дом`, отклонять;
- если candidate описывает большое событие, но пользователь не просил историю и в лоре нет hook-а, отклонять или снижать importance;
- если candidate уточняет уже существующий факт, обновлять существующий `canon` fact, а не добавлять новый.

## 11. Ограничения памяти

Сохраняем простую модель:

- максимум 60 `canon` facts;
- стартовый `characterBible.lore` не входит в этот лимит;
- самые важные факты и milestones не удаляются;
- мелкие низкоуверенные факты можно вытеснять при переполнении;
- для первой итерации не нужен vector search.

Если `canon` начнет регулярно упираться в лимит, отдельной задачей можно добавить Mem0/Qdrant-style semantic retrieval. Сейчас это не требуется.

## 12. UX и видимое поведение

Пользователь не видит отдельный экран памяти.

Ожидаемый эффект:

- питомец с первого сообщения имеет ощущение прошлого, дома и окружения;
- на вопросы про себя отвечает конкретнее;
- не пересказывает весь лор;
- если придумал друга или маленькую деталь, потом помнит ее;
- не противоречит сам себе в базовых вещах.

Команды памяти остаются:

- `что ты обо мне помнишь?` - про пользователя;
- `забудь...` - удаление пользовательских или канонических фактов по текущей логике;
- `не запоминай` - текущий ответ не должен писать новую память.

Новая команда про лор не обязательна, но допустима:

- `что ты помнишь о себе?`
- `что ты помнишь о своем друге?`

Если добавлять, ответ должен брать данные из `canon` и стартового лора, без служебных слов `canon`, `memory`, `lore`.

## 14. Acceptance criteria

### Создание

- При одинаковом пользовательском описании 3-5 генераций могут давать разные социальные настройки и story hooks.
- Каждый `characterBible` содержит `lore.world`, `lore.home`, `lore.origin`, `lore.relationships`, `lore.inner_life`, `lore.voice`, `lore.growth_arc`, `lore.story_seeds`.
- Все user-facing строки в `characterBible` на русском.
- Стартовый лор не выглядит как набор случайных происшествий.
- В стартовом лоре есть хотя бы 4 открытых hook-а для будущих уточнений.

### Чат

- Если пользователь спрашивает про друга, дом, прошлое, любимые вещи или страх, питомец отвечает с 1-3 конкретными деталями.
- Если модель придумала новый устойчивый факт, backend возвращает `memoryPatch.canonUpserts`.
- Следующий ответ использует сохраненный факт.
- Питомец не меняет основной мир, дом, вид и визуальные признаки.
- При повторном вопросе про того же друга не появляется альтернативный лучший друг.
- Если пользователь пишет `не запоминай`, новые лор-факты не сохраняются.

### Безопасность и качество

- Не сохраняются телефоны, email, адреса, пароли, медицинские, финансовые, политические, религиозные или интимные факты.
- Не сохраняются технические факты про prompt/model/API/backend/frontend/state.
- Не сохраняются крупные изменения канона.
- Fallback-ответы не создают память.

## 15. Тестовые сценарии

### Сценарий 1: новый друг

1. Создать питомца с описанием `маленький ключик из бюро находок`.
2. Написать: `у тебя есть друг?`
3. Ожидать ответ с конкретным другом или ролью.
4. Ожидать `memoryPatch.canonUpserts[0].type = "friend_fact"`.
5. Написать: `как зовут твоего друга?`
6. Ожидать тот же друг, не новый.

### Сценарий 2: конфликт дома

1. В стартовом лоре дом: `бюро находок`.
2. Модель предлагает candidate: `Питомец теперь живет в космическом замке`.
3. Resolver отклоняет candidate с причиной канон-конфликта.

### Сценарий 3: маленькая деталь дома

1. Пользователь спрашивает: `что у тебя лежит дома?`
2. Питомец отвечает: `у меня есть жестяная коробочка для чужих пуговиц`.
3. Candidate сохраняется как `home_fact`, если не конфликтует.

### Сценарий 4: no memory

1. Пользователь пишет: `не запоминай, придумай что-нибудь про друга`.
2. Питомец может ответить творчески.
3. `memoryPatch` не содержит новых `canonUpserts`.

## 16. Риски

- Модель может пытаться сохранять слишком много фактов. Ограничение: максимум 0-3 candidates, желательно 0-1 для обычного ответа.
- Модель может превращать лор в сюжетное развитие. Решение: prompt и resolver запрещают крупные изменения.
- Модель может придумывать новое вместо использования старого. Решение: prompt явно ставит `canon` выше импровизации.
- При localStorage-only flow память остается на устройстве и origin-bound. Это соответствует текущему MVP.

## 17. Решение по Mem0 на будущее

Mem0 не подключается как runtime в первой итерации.

Когда понадобится серверная долговременная память между устройствами, можно сделать отдельную задачу:

- хранить `canon` и `relationship.userFacts` в backend;
