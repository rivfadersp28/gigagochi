# ТЗ — визуальные режимы питомца: normal / sad / happy

## 1. Цель

Добавить полноценную счастливую версию сцены питомца: статичный ассет и зацикленное видео с лёгкой улыбкой и морганием. Переключать normal / sad / happy автоматически по трём параметрам питомца и вручную из Debug Panel.

Фича применяется только к основному экрану питомца. Текстовые ответы, `pet.mood`, расчёт параметров и механика их изменения не входят в задачу.

## 2. Текущее поведение — факты из кода

- Параметры питомца: `hunger`, `happiness`, `energy`, диапазон `0–100`; локально значения могут быть дробными.
- Красный индикатор включается при значении строго `< 30`.
- Зелёный индикатор включается при значении `>= 70`.
- Грустная сцена включается, если хотя бы один параметр `< 30` и готовы отдельные sad-ассеты.
- Normal-сцена имеет `videoUrl`; sad-сцена имеет отдельные изображение и `sadVideoUrl`.
- Sad-изображение создаётся в два edit-прохода: сначала новая поза, затем восстановление композиции по исходной сцене.
- Sad-видео генерируется из итогового sad-изображения.
- Текущая Debug Panel умеет только принудительно включить/выключить sad-ассет.
- `pet.mood` рассчитывается по другим правилам и не совпадает с требуемыми visual modes. Использовать его как источник нового визуального режима нельзя.

## 3. Термины

| Термин | Определение |
|---|---|
| `normal` | Исходная сцена и исходное видео питомца |
| `sad` | Сцена с сидящим плачущим питомцем и отдельное видео плача |
| `happy` | Сцена с тем же питомцем и фоном, но с лёгкой улыбкой; видео сохраняет улыбку и добавляет обычное моргание |
| `auto mode` | Режим, выбранный автоматически по текущим параметрам |
| `debug override` | Временный ручной выбор visual mode, не изменяющий параметры питомца |

## 4. Состав интерфейса

| Элемент | Обязательный | Описание |
|---|---:|---|
| Кнопка `Грустный` | Да | Принудительный preview sad-ассетов |
| Кнопка `Нормальный` | Да | Принудительный preview normal-ассетов |
| Кнопка `Счастливый` | Да | Принудительный preview happy-ассетов |
| Selected-state кнопки | Да | Показывает активный `debug override` |
| Индикатор `Авто` | Да | Показывает, что override отсутствует и режим управляется параметрами |
| Прогресс фоновой генерации | Да | Отражает готовность sad и happy image/video |

Три кнопки разместить одной segmented-группой в Debug Panel вместо текущей кнопки «Сделать грустным».

## 5. Правила выбора visual mode

### 5.1. Автоматический режим

Проверки выполняются в следующем порядке:

1. Если `hunger < 30` **или** `happiness < 30` **или** `energy < 30` → `sad`.
2. Иначе, если `hunger >= 70` **и** `happiness >= 70` **и** `energy >= 70` → `happy`.
3. Иначе → `normal`.

| Значения параметров | Результат |
|---|---|
| Хотя бы один параметр `29.999` или ниже | `sad` |
| Минимальный параметр ровно `30` | Не `sad`; далее проверяется `happy`, иначе `normal` |
| Все три параметра ровно `70` или выше | `happy` |
| Хотя бы один параметр от `30` включительно до `70` не включительно | `normal` |

Порог `>= 70` выбран намеренно: он совпадает с текущим порогом зелёного индикатора. Формулировку «больше 70%» в исходной постановке трактовать как «зелёная зона».

### 5.2. Приоритет и fallback

- `debug override` имеет приоритет над автоматическим режимом.
- Если для выбранного `sad` или `happy` не готовы одновременно изображение и видео, показывать `normal`.
- Частично готовую пару (только изображение или только видео) не показывать.
- Ошибка генерации sad не должна блокировать генерацию happy, и наоборот.
- Ошибка любой производной версии не должна ломать готовую normal-версию.
- `pet.mood` продолжает использоваться в существующих диалоговых сценариях, но не выбирает media asset главного экрана.

## 6. Генерация happy-версии

### 6.1. Общая схема

После готовности исходных normal image/video запустить независимый happy-пайплайн:

1. Первый image edit исходной normal-сцены → временный happy-вариант.
2. Второй image edit из двух референсов:
   - первая картинка — исходная normal-сцена, эталон фона, камеры, масштаба и композиции;
   - вторая картинка — временный happy-вариант, эталон лёгкой улыбки.
3. Сохранить итоговое happy-изображение.
4. Сгенерировать happy-видео из итогового happy-изображения.

Оба производных пайплайна — sad и happy — всегда используют исходную normal-сцену. Happy нельзя строить от sad и sad нельзя строить от happy.

### 6.2. Первый prompt happy image edit

```text
пусть персонаж слегка приподнимет голову, чуть более жизнерадостно посмотрит и слегка естественно улыбнётся
больше ничего не меняй

КРИТИЧЕСКИ ВАЖНО СОХРАНИТЬ КОМПОЗИЦИЮ: используй точно ту же камеру, дистанцию до персонажа, кадрирование, перспективу и размер персонажа в кадре. Не приближай персонажа, не увеличивай голову или тело, не делай портретный или крупный план. Сохрани точное положение персонажа, фон, освещение, цвета, дизайн, одежду и все предметы. Измени только голову и выражение лица: чуть-чуть приподними подбородок, сделай глаза немного более открытыми и живыми, сохранив их исходную форму, и добавь лёгкую закрытую улыбку. Не запрокидывай голову, не меняй положение тела, не открывай рот, не показывай зубы и не превращай эмоцию в широкую или мультяшную.
```

### 6.3. Второй prompt happy composition refinement

```text
Первая картинка — единственный обязательный эталон композиции, масштаба и окружения. Сохрани из первой картинки точную камеру, кадрирование, перспективу, фон, положение персонажа, размер головы в пикселях и общий размер персонажа в кадре. Вторая картинка используется только как референс слегка приподнятой головы и более жизнерадостного выражения персонажа.

Верни сцену в композиции первой картинки. Персонаж должен находиться на той же дистанции от камеры, сохранять положение тела и занимать ровно столько же места, сколько на первой картинке. Сохрани дизайн персонажа, одежду, аксессуары, фон, освещение и все предметы первой картинки. Перенеси со второй картинки только чуть приподнятый подбородок, немного более открытые и живые глаза и лёгкую естественную закрытую улыбку. Не изменяй размер или форму головы и глаз, не открывай рот, не показывай зубы и не меняй ничего больше.
```

Промпт намеренно не перечисляет конкретные аксессуары вроде посоха, книги или флаконов: он должен работать с любым сгенерированным персонажем.

### 6.4. Happy video

- Источник: итоговое happy-изображение после второго edit-прохода.
- Персонаж сохраняет лёгкую улыбку без движения рта, головы или тела.
- Единственная обязательная анимация — естественное моргание, идентичное normal-видео.
- Для happy использовать тот же video prompt, что и для normal, либо одну общую константу без расхождения копий.
- Камера, фон, свет, цвет, фокус, поза, одежда и предметы должны оставаться статичными.
- Текущая длительность ролика сохраняется: `4,000 ms` и бесконечный повтор на клиенте.

Рекомендуемые имена файлов:

```text
teen-happy-pose.png   # временный, удалить после refinement
teen-happy.png
teen-happy.mp4
```

## 7. Контракт ассетов и статусы генерации

### 7.1. Asset set

- Добавить `happyVideoUrl?: string`.
- После готовности happy-изображения записать его в `images[stage].happy` для всех текущих stage fallback, аналогично sad.
- `images[stage].idle` остаётся normal poster.
- `images[stage].sad` остаётся sad poster.
- `images[stage].hungry` не используется для выбора visual mode в рамках этой задачи.

### 7.2. Фазы background generation

Поддержать как минимум:

```text
generating_sad_image
generating_sad_video
generating_happy_image
generating_happy_video
completed
```

Normal image/video остаются минимально необходимым результатом, после которого главный экран можно открыть. Sad и happy догенерируются в фоне.

Статусы и ошибки sad/happy должны храниться раздельно либо позволять однозначно определить готовность каждой пары. Один общий `backgroundGenerationError`, из-за которого прекращается вся цепочка после первой ошибки, требованиям не соответствует.

### 7.3. Готовность режима

| Режим | Условие готовности |
|---|---|
| `normal` | Есть `images[stage].idle`; `videoUrl` опционален по текущему fallback |
| `sad` | `images[stage].sad` отличается от idle и существует `sadVideoUrl` |
| `happy` | `images[stage].happy` отличается от idle и существует `happyVideoUrl` |

## 8. Debug Panel

- Заменить boolean `isSadAssetForced` на `visualModeOverride: "sad" | "normal" | "happy" | null`.
- Нажатие на одну из трёх кнопок устанавливает соответствующий override.
- Повторное нажатие на выбранную кнопку снимает override и возвращает `auto mode`.
- `normal` должен действительно принудительно показывать normal даже при красных или зелёных параметрах.
- Override не изменяет `stats`, `pet.mood`, `localStorage`, backend snapshot или generated assets.
- Override живёт только в состоянии текущего dashboard и сбрасывается после перезагрузки/размонтирования.
- `Грустный` disabled, пока не готовы sad image/video.
- `Счастливый` disabled, пока не готовы happy image/video.
- `Нормальный` доступен всегда.

## 9. Переключение media assets

- `normal` → poster `images[stage].idle`, video `videoUrl`.
- `sad` → poster `images[stage].sad`, video `sadVideoUrl`.
- `happy` → poster `images[stage].happy`, video `happyVideoUrl`.
- Перед заменой активного источника preload одновременно нового poster и video.
- Пока новая пара загружается, продолжать показывать предыдущую сцену; не возвращать весь dashboard в initial loader.
- После загрузки заменить poster и video согласованно, чтобы не появлялась комбинация изображения одного режима и видео другого.
- При смене `src` перезапустить видео с начала тем же способом, который используется сейчас.
- Отдельный crossfade или новая UI-анимация в задачу не входят.

## 10. Движение и анимация

| Триггер | Свойства | Длительность | Кривая |
|---|---|---:|---|
| Запуск normal/happy video | Только веки; естественное моргание, остальной кадр статичен | `4,000 ms` media loop | Не задаётся UI-кривой; движение генерирует video model |
| Запуск sad video | Только слёзы и минимальное дрожание век; остальной кадр статичен | `4,000 ms` media loop | Не задаётся UI-кривой; движение генерирует video model |

Параметры video generation, которые должны оставаться конфигурируемыми: `duration`, `resolution`, `aspectRatio`, normal/happy prompt и sad prompt. Точный момент моргания внутри generated video сейчас недетерминирован и не может быть выражен через `productive`/`expressive` без перехода на программную анимацию.

## 11. Ограничения

- Production rollout временно разрешён только Telegram ID `62943754`.
  Backend не запускает derived generation для остальных пользователей, а
  frontend независимо принудительно показывает им normal даже при наличии
  старых sad/happy URL в localStorage.
- Happy-версия отличается от normal только лёгкой улыбкой и морганием в видео.
- Нельзя менять фон, композицию, масштаб, позу, дизайн, одежду, аксессуары или количество предметов.
- Улыбка закрытая и сдержанная: без зубов, открытого рта и преувеличенной эмоции.
- Нельзя использовать happy scene как reference для генерации историй вместо текущего выбранного экранного ассета; существующее правило выбора current on-screen asset сохраняется.
- Старые asset sets без `happyVideoUrl` должны продолжать работать через normal fallback.

## 12. Проверяемые сценарии

| Сценарий | Ожидаемый результат |
|---|---|
| `29 / 100 / 100` | `sad` |
| `30 / 100 / 100` | `normal` |
| `70 / 70 / 70` | `happy` |
| `100 / 70 / 100` | `happy` |
| `100 / 69.9 / 100` | `normal` |
| Авто выбрал sad, debug выбран `Нормальный` | Показывается normal |
| Авто выбрал normal, debug выбран `Счастливый` | Показывается happy |
| Happy poster готов, video отсутствует | Показывается normal |
| Happy generation упала, sad готов | Normal/sad продолжают работать; ошибка happy видна в debug progress |
| Переключение normal → happy | До полной загрузки happy poster/video остаётся normal; затем оба источника меняются вместе |

## 13. Риски и открытые вопросы

- Image model может менять лицо или фон даже при refinement-проходе; это ограничение текущего генеративного подхода.
- Два image edit и одно video generation на happy увеличат стоимость и общее время фоновой генерации примерно на стоимость ещё одной sad-подобной ветки.
- Точная синхронизация и частота моргания между normal и happy не гарантируется моделью. Требование трактуется как одинаковый video prompt и одинаковые настройки генератора.
- Нужно решить, показывать ли пользователю частичную готовность sad/happy в production UI; в текущем объёме прогресс остаётся только в Debug Panel.
