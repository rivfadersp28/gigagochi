# Background Story Generator Plan

## Goal

Add background fantasy events for a pet. In the first debug version an event is
generated by the Telegram bot command `/story`. Later the same service can be
called by a scheduler N times per day.

Generated events are saved into the pet's existing personal story RAG store, so
they can be recalled in normal dialogue.

## Architecture Decision

Do not create a new parallel memory store for now. There is no product logic yet
that requires a separate events domain such as unread/read status, timeline UI,
expiry, stat consequences, or per-day quotas per event type.

Store generated background events in:

```text
characterBible.extensions.story_library_overlay.bricks
```

Use a dedicated brick pool:

```json
{
  "pool": "events",
  "poolLabel": "Фоновые события",
  "attributes": {
    "eventType": "attack",
    "generatedBy": "background_story"
  }
}
```

This reuses the existing RAG path: `story_library_overlay` is already merged
with the global story library and searched before global bricks.

## Important Sync Constraint

Telegram bot can see only the backend push snapshot. Normal Mini App chat uses
the pet from browser `localStorage`.

If `/story` saves an event only into the backend snapshot, the regular Mini App
chat will not see it in RAG, and the next `/api/push/snapshot` can overwrite it.

Therefore the implementation must sync server-generated story bricks back into
frontend localStorage.

## Backend Plan

1. Add `backend/app/services/background_story_service.py`.

2. Build a compact character dossier for the prompt:
   - name
   - description
   - stage
   - semantic state params from `stateLayer.stateParamLabels`
   - identity/species/signature
   - visual features
   - inner state
   - lore/world/home/routines/relationships/story seeds
   - lite overlay facts
   - recent story overlay bricks
   - now/timezone

3. Do not pass full raw `characterBible` or asset URLs into the story prompt.
   This keeps the prompt smaller and avoids repeating the previous heavy-prompt
   failure mode.

4. Add configurable prompt fields to `backend/data/speech_runtime.json`:

```json
{
  "backgroundStory": {
    "systemPrompt": "...",
    "userTemplate": "Придумай фэнтези историю как на этого персонажа напали. Вот описание персонажа и состояние: {character}",
    "defaultEventType": "attack",
    "maxStoryChars": 1200,
    "maxRagChars": 500
  }
}
```

5. Extend `backend/app/services/pet_reply_engine/speech_runtime.py`:
   - validate required `backgroundStory` keys
   - expose `background_story_system_prompt()`
   - expose `background_story_user_prompt(values)`

6. Generate structured JSON:

```json
{
  "title": "string",
  "summary": "string",
  "storyText": "string",
  "eventType": "attack",
  "valence": "negative",
  "tags": ["string"],
  "ragText": "string"
}
```

7. Convert the result into a story-library patch:

```json
{
  "version": 1,
  "bricks": [
    {
      "source": "pet_overlay",
      "pool": "events",
      "poolLabel": "Фоновые события",
      "name": "{title}",
      "text": "{ragText}",
      "attributes": {
        "summary": "{summary}",
        "fullStory": "{storyText}",
        "eventType": "attack",
        "valence": "negative",
        "tags": [],
        "generatedBy": "background_story"
      }
    }
  ]
}
```

8. Save the patch with existing `merge_story_library_patch()`.

## Push Snapshot / Frontend Sync

1. Add a public service method in `telegram_push_service.py`, for example:

```python
generate_story_for_telegram_user(telegram_id: int) -> dict[str, Any]
```

2. It should:
   - find the user's snapshot
   - compute current pet state with lazy stat decay
   - generate a background story
   - merge the story patch into
     `record.pet.characterBible.extensions.story_library_overlay`
   - save the updated record
   - return the story and `storyLibraryPatch`

3. Update `register_push_snapshot()` so new frontend snapshots preserve
   server-generated `story_library_overlay` bricks from the previous record.

4. Extend `LocalPetPushSnapshotResponse`:

```python
storyLibraryPatch: dict[str, Any] | None = None
```

5. Update frontend `registerPetPushSnapshot()` and `PetDashboard` to apply:

```ts
localPet.applyStoryLibraryPatch(response.storyLibraryPatch)
```

## Telegram Bot Debug Command

1. Add `/story` handling in `backend/app/bot.py`.

2. Flow:
   - no snapshot: reply that the user must open the Mini App first
   - success: send a compact generated story and Mini App button
   - provider failure: send a short debug-safe error without stack trace

3. Bot should call the backend service, not read/write the JSON store directly.

## RAG Changes

1. Add `events` to `context_assembler.POOL_KEYWORDS`.

2. Suggested keywords:

```text
событие, случилось, история, приключение, напали, атака, фон, ночью, вчера
```

3. Keep `pet_overlay` priority above global story bricks.

4. If future chat extraction should save events too, extend
   `STORY_LIBRARY_EXTRACTION_SCHEMA` pool enum with `events`.

## Admin

1. Add a dialogue influence item in `local_admin_store.py`:
   - id: `background_story_prompt`
   - label: `Фоновые истории`
   - surfaces: `["background_story"]`
   - source: `speech_runtime`
   - editable: `true`
   - configPath: `backgroundStory`

2. Add a section in `SpeechAdmin.tsx` for:
   - system prompt
   - user template
   - default event type
   - max story chars
   - max RAG chars

3. Keep save/deploy through the existing `speech_runtime.json` admin flow.

## Later Scheduler

After `/story` works, add scheduler settings:

```text
BACKGROUND_STORY_ENABLED
BACKGROUND_STORY_MAX_PER_DAY
BACKGROUND_STORY_INTERVAL_SECONDS
BACKGROUND_STORY_MIN_INTERVAL_HOURS
```

The scheduler must call the same service as `/story`.

## Tests

Backend:

- background story service creates a valid `events` story-library patch
- generated patch is merged into push snapshot
- `register_push_snapshot()` preserves server-generated story bricks
- `speech_runtime` validates `backgroundStory`
- bot `/story` handles missing snapshot

Frontend:

- `registerPetPushSnapshot()` accepts `storyLibraryPatch`
- `PetDashboard` applies `storyLibraryPatch` into local pet state

## Risks

- If server-generated stories are not synced back to localStorage, normal chat
  RAG will not see them.
- Full raw `characterBible` in the prompt can make generation heavy and brittle.
- A separate events store is premature until there is product logic that needs
  timeline status, read state, expiry, stat effects, or UI filtering.
