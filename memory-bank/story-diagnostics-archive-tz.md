# ТЗ: архив историй и полных ответов модели

Статус: отложено, реализация не начата.

## Цель

Сохранять данные генерации историй на сервере для последующего анализа ошибок и качества:

- полные ответы модели, включая ответы, отклонённые валидатором;
- технические повторные попытки;
- полные состояния интерактивных путешествий, которые сейчас хранятся в `localStorage` пользователя;
- историю изменений путешествия, а не только финальное состояние.

Обычные ответы чата в этот архив не входят.

## Текущее состояние

- `backend/logs/ai-responses.jsonl` сохраняет метаданные ответа модели, но не полный ответ.
- Завершённое интерактивное путешествие отправляется на сервер.
- Незавершённые и ошибочные состояния путешествия остаются только в браузерном `localStorage`.
- `finale.json` сохраняется внутри публичного `/static/generated` и содержит данные владельца, включая Telegram ID. Этот файл необходимо перенести в приватное хранилище. Публичными должны оставаться только медиафайлы PNG/MP4.

## Серверный архив

Создать приватный архив по пути `/app/data/story-archive` на отдельном постоянном Docker volume.

Архив не должен раздаваться через `/static`.

Для каждого ответа модели, относящегося к генерации истории, сохранять:

- полное содержимое ответа модели;
- тип генератора и этап;
- `travelId`, если применимо;
- номер части путешествия, если применимо;
- Telegram ID владельца;
- имя модели и провайдера;
- время запроса и ответа;
- признак первичной или технической повторной попытки;
- результат обработки: принят, отклонён или завершился ошибкой;
- код и сообщение валидатора при отклонении;
- идентификатор запроса провайдера, если он доступен;
- usage/cost, если провайдер их вернул.

Полные промпты по умолчанию не сохранять. Их хранение должно включаться отдельно, поскольку промпты могут содержать память пользователя и персональные данные.

Архивировать ответы следующих генераторов:

- интерактивное путешествие;
- обычное путешествие;
- фоновые истории;
- полные истории.

Не архивировать обычный чат, ambient, proactive и операции памяти в рамках этого ТЗ.

## Синхронизация `localStorage`

Добавить авторизованный API endpoint для синхронизации полного `LocalInteractiveTravel`.

Frontend должен отправлять состояние:

- после каждого успешного изменения `localStorage`;
- после восстановления существующего путешествия из `localStorage`;
- перед локальным сбросом путешествия;
- после завершения путешествия.

Синхронизация должна быть best-effort и не должна блокировать интерфейс или прохождение истории.

На сервере хранить отдельные ревизии состояния, привязанные к `travelId`, а не только последний snapshot. Повторная отправка одной ревизии должна быть идемпотентной.

Сервер обязан проверять, что авторизованный Telegram-пользователь является владельцем переданного путешествия.

## Приватность и срок хранения

- Текстовые архивы и данные владельца не должны попадать в публичную статику.
- Доступ к архиву разрешён только через локальную админку или защищённый административный API.
- Не выводить полные ответы модели в обычные production-логи контейнера.
- Предварительный срок хранения — 90 дней, с настройкой через backend config/env.
- Перед реализацией определить правила удаления архивов и связанных с ними персональных данных.

## Миграция существующего хранения

- Перенести сохранение `finale.json` из `/app/static/generated/<travelId>/` в приватный архив.
- Оставить PNG/MP4 в `static/generated`.
- Не ломать существующую лабораторию финальных видео: перевести её чтение финала на новый приватный путь.
- Отдельно решить, нужно ли переносить уже сохранённые production-файлы `finale.json`.

## Проверки

Добавить тесты на следующие сценарии:

- сохраняется полный первичный ответ, отклонённый валидатором;
- сохраняется полный ответ технической повторной попытки;
- принятый и отклонённый ответы различимы в архиве;
- незавершённое путешествие синхронизируется с сервера и не теряется после локального сброса;
- повторная отправка snapshot не создаёт дубликат ревизии;
- пользователь не может записать состояние в чужое путешествие;
- `finale.json` больше не доступен через `/static`;
- недоступность архива не ломает пользовательский сценарий генерации.

## Ограничения реализации

- Не смешивать архив историй с `ai-responses.jsonl` и публичными generated assets.
- Не менять поведение генераторов и валидаторов в рамках этого ТЗ.
- Не перезаписывать существующие незакоммиченные изменения в `schemas.py`, `interactive_travel_service.py`, `localInteractiveTravel.ts` и связанных файлах.
- После принятия архитектуры обновить `memory-bank/architecture.md` и `memory-bank/gotchas.md` только по фактически реализованным решениям.
